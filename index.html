What is Maker Pattern

Maker pattern has the goal of manipulating the genes of your objects and classes to add useful tools, features, properties or apply more complex patterns.

How util is that?

There are so many useful patterns out there, you probably already known most of them,
but why to apply each one every time you need?
"If it is a pattern to be repeated, then it should be applied by a machine!"
There are many advantages, such as:
  • A lot less code for you to type
  • The API for that pattern will always be the same(enough memorising APi's methods that do the same, in different projects)
  • Once maker is updated, so will be all your objects that use maker! You don't need to write a single extra line to adapt of your objects to have that useful feature that was just added to some design-pattern
  • Creating a framework? Apply the best methodologies and patterns with maker, to your exposed objects
  • Building a simple page, but when a json changes, you want the interface to adapt...what about less than 10kb of js and one line to apply it?
  • Really easy to learn how to use, and then, lean design-patterns through it
  • Apply advanced patterns and techniques with one line
  • It's open source...contribute creating new makers and applying patterns.
HOW
Each maker may offer different options and possibilities, but you only have to understand how to use maker in two ways.
They use javascript wrappers wo apply the extra features to objects and classes.
No native prototype is changed.
  • Making Objects/Instances
    By passing an object(from, json, raw object or instance, for example), the object itself is engineered:
    Example: 
    
	make.observable(myObject);
  • Making Functions/Classes
    By passing a function. In this case, as functions are not passed by reference, you will received the changed function returned from the maker
    
	var fn= make.debounce(function(){...});
	window.onscroll= fn;
To see how each maker works, check some demos at out samples page.

Here, some teasing examples (they work for classes as well):
Observable:
// making an object observable
var o = {
  name: 'felipe',
  age: 28
};

// then make it observable
make.observable(o);

// now you can listen to it like this:
o.on('name', function(newValue){
  alert("someone changed o's name to " +
        newValue);
});

// everytime any script does
// something like this:
o.name= 'other name';
// or
o.setName('another name');
// or this
o.set('name', 'yet another name');

// you will see it in your console.
Tieable:
// making an object tieable
var o = {
  name: 'felipe',
  age: 28
};

// then make it tieable
make.tieable(o);

// now we tie it to someone
var el= document.getElementById('foo');
o.tie('name', foo, 'innerHTML');

// from now on, whenever some script does
o.name= 'other name';
// or
o.setName('another name');
// or even
o.set('name', 'yet another name');
// the tied element property will change
// in this case, foo.innerHTML
Worker:
// making a worker from a function
// let's say you have an expensive
// or heavy function
var func= function(arg){
	var i= 0;
	for(; i < 10000000; i++){
        // it takes too long to end
    }
    return 'ended ' + arg;
};
// you can make it run as a worker
func= make.worker(func);

// now, you run them as threads
console.log(func('A'));
console.log(func('B'));
console.log(func('C'));
console.log(func('D'));
console.log(func('E'));
console.log(func('F'));
// they will run simutaneously and
// may even finish in a different
// order
Debounce/Throttle:
// making function debounce or throttle
// functions that access offset values
// may be very slow if called repeatedly
function showOffset(){
	console.log(
		document.body.offsetWidth;
	);
}

// so, instead, we debounce it
showOffset= make.debounce(showOffset);

// this way, it will only execute after
// the user stops resizing the window
window.onresize= showOffset;
// throttle, would execute the showOffset
// for each interval of time
